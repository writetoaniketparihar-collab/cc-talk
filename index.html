Part 1 of 3

First up: CLAUDE.md

One file that tells Claude who you are.

What is a CLAUDE.md?

1
It's a Markdown file. Plain text with a few simple symbols for formatting: a # for a heading, a - for a bullet. That's all ".md" means. Opens in anything, nothing to install.
2
Claude Code reads it automatically. CLAUDE.md is the one file it loads first, at the start of every new chat. You never attach it or paste it in.
3
It lives in a folder on your computer. Mine sits at /Users/aniket/CLAUDE.md, right in my home folder. Claude looks there and finds it on its own.
4
You can have more than one. A global file for everything, plus one inside any project. Claude reads all that apply.
None of this is new. READMEs have gotten new people up to speed for 20 years, and CLAUDE.md does the same, for Claude.

My own CLAUDE.md.

Watch for three things: it's short, every line is a fact about me, and it tells Claude how I like to work.

So why write one?

1
Every new chat starts blank. Claude forgets who you are, what you do, and how you like things.
2
So you re-explain yourself. The same background, typed again in every single chat.
3
Write it once, and you never repeat yourself again. Every chat starts knowing you, and answers come out yours, not generic.
That's the whole point. Say it once, never again.

A template to start from.

CLAUDE.md (template)
# Who I am
- your role and team
- the company, and what you own
- "keep it plain" if that's you

# What I work on
- your product, in one line
- who its users are
- who you work with
# How I write
- your tone for docs
- the format you like
- words you never want to see

# How to work with me
- give me options, not essays
- ask what's missing first
- push back when I'm wrong
Four sections. Fill in your own, keep it under a page.

What goes in. What stays out.

Goes in
  • Your role, and what you own
  • Your product, and who it's for
  • How you want things written
  • How Claude should work with you
Stays out
  • Vague fluff like "be strategic"
  • Your full resume or life story
  • A one-time task, like today's email
  • Passwords or anything secret
Part 2 of 3

Next up: Skills

The file that gets a repeated task done your way.

What is a skill?

1
Another Markdown file. Same plain text as CLAUDE.md. This one is called SKILL.md, and it describes one task: what it is, when to use it, and the steps.
2
It lives in a skills folder. Mine is at /Users/aniket/.claude/skills.
3
Claude picks it up by itself. Each skill starts with a line saying when to use it. Ask for that task, and the skill kicks in. You never have to remember it exists.
4
Make one per repeated task. The PRD, the Friday update, the interview debrief. Each gets its own file, and only the one you need loads.

My own skill.

Watch for three things: it does one job, the description says when to use it, and every step is checkable.

So why write one?

1
Some tasks come back every week. The PRD, the status update, the interview debrief.
2
So you re-explain the how, every time. Your template, your tone, your steps, typed again in every chat.
3
Write it once, and you never re-explain again. You say "draft the PRD" and it comes out your way, every time.

Now, let's build one.

One sentence to Claude, and it writes the skill for me. I'll build a "write my weekly report" skill that turns rough notes into an organized report.

A template to start from.

skills/your-task/SKILL.md (template)
name: one or two words, e.g. "prd"
description: what it does, in one line.
  Then "Use when..." in the words you'd actually say

# Steps
1. the steps in order, each one checkable
2. "start with a summary" beats "make it engaging"
3. end with what the finished output looks like

What goes in. What stays out.

Goes in
  • Just one task, like "draft my weekly report" *
  • The everyday words you'd say to trigger it *
  • How you want the task done, step by step *
  • What you want at the end, like an email or a one-pager
  • Anything it must never break, like "always under a page"

* the must-haves, the rest is optional

Stays out
  • A second task (that's a new skill of its own)
  • Fuzzy instructions like "make it good"
  • Anything you'd do differently each time

What could yours be?

A skill is any task you do the same way every week. A few, by job:

Product manager
  • Draft a PRD from rough notes
  • Write the weekly update
  • Turn interview notes into themes
Sales & marketing
  • Turn a call into a follow-up email
  • Write a LinkedIn post from rough notes
  • Fill a proposal from your template
Customer service
  • Reply to a ticket in your tone
  • Summarize a long complaint thread
  • Write a help doc from a fix

Tip: let Claude write it for you. Ask, "What do I keep repeating?" Pick the ones you do the same way each time, then tell Claude, "please turn them into skills."

Part 3 of 3

Last up: Hooks

The check that runs on its own, every single time.

What is a hook?

1
It's a tiny script that Claude Code runs for you.
2
It does one of two things: a task for you, like sending a desktop notification when Claude finishes, or blocking a mistake, like stopping a bad prompt before it goes in.
3
The hook sits in a hooks folder, mine is at /Users/aniket/.claude/hooks, and a separate settings file tells Claude Code when to run it.

My own hook.

Watch for three things: it runs on every prompt I send, the script is dead simple, and I gave myself a way past it.

So why write one?

1
A written rule can be missed. The rules in your CLAUDE.md and skills are ones Claude reads and remembers, most of the time, but not always.
2
A hook can't be missed. A hook is a script, not a rule Claude has to remember, so it just runs on its own, every single time.
3
So use it for anything that should always happen. Block a mistake, like customer data slipping into a prompt. Or do something helpful, like a notification after Claude Code finishes the work.
Hooks… ensure certain actions always happen.Claude Code hooks documentation

Now, let's build one.

Same idea, for a hook. I'll build one that blocks any prompt with an email or phone number, the personal data you should never paste into AI, then watch it stop me in real time.

A template to start from.

the hook: When / Then
When: the moment it fires

Then: what happens
A ready example. Save it to your hooks folder, then ask Claude to switch it on.

What goes in. What stays out.

Goes in
  • A clear trigger, when it should run, like "when Claude finishes"
  • Only one job: stop a mistake, or do a task
  • If it blocks something, the check must be a simple yes or no, like "is there a customer name in it?"
  • If it blocks something, what to fix and a way past it
Stays out
  • Several jobs crammed into one
  • A judgment call, like "does this sound good?"
  • A block with no way past it

What could yours be?

A hook is one dumb check on the mistake you can't afford. A few, by job:

Product manager
  • Block any prompt with a customer name
  • Stop a PRD saving with no success metrics
  • Bounce a spec over your page limit
Sales & marketing
  • Block a post that names unreleased pricing
  • Stop an email with a competitor's name
  • Bounce a LinkedIn post over the character limit
Customer service
  • Block a reply with an internal note in it
  • Stop a message with a full card number
  • Bounce a reply missing the ticket number
